# Spike 3: 传历史对话列表给新会话（用户真实诉求，不碰 shouldQuery）

**日期**：2026-06-01
**动机**：spike1/2 在 `shouldQuery:false` 上钻牛角尖（lead 引入的伪需求"让历史不触发逐条回应"）。用户连续质疑「你列表传过去 SDK 没回复你吗」「别管 shouldQuery 会死」——点醒 lead：真实诉求只是「把 DB 历史对话当 message 传给新会话让它有上下文」，根本不需要 shouldQuery。本 spike 回到原点直接测。

**结论先行**：✅ **可行，且揭示最佳载体**。传历史给新会话 SDK 正常回复且看到上下文（推翻 spike1/2 给人的"不可行"错觉——那是 shouldQuery 的锅，不是"传历史"的锅）。**关键实测**：历史组织成「**1 条带角色标注的结构化文本**」上下文保留最完整；拆成「多条独立 message」反而丢中间上下文。

## 实测结果

| 变体 | 历史组织方式 | num_turns | 历史可见 | 关键观察 |
|---|---|---|---|---|
| **3 方式A** | 整段历史塞 **1 条 user message**（`[历史回顾]\n用户:...\n助手:...`）+ 当前问题 1 条 | 1 | **小明✅ 紫✅ 旺财✅ 全可见** | SDK 完整看到结构化历史文本，正常回答 |
| **3b 方式B** | 历史**每条独立成 message**（真·多条列表）+ 当前问题 | 1 | 小明✅ 紫✅ **旺财✗ 丢失** | 多条独立时中间消息被挤掉/只保留最近，且多 turn |

## 铁证分析

1. **「传历史列表」本身完全可行**（推翻 spike1/2 的"不可行"错觉）：spike1/2 的 `num_turns=0` 失败**100% 是 `shouldQuery:false` 的锅**（spike2b 隔离实验确认），与"传历史"无关。去掉 shouldQuery，传历史 SDK 正常工作。
2. **单条结构化文本 > 多条独立 message**（方式A vs 3b 实测）：
   - 方式A（1 条文本含全部历史）：全部历史可见 ✅
   - 方式B（多条独立 message）：中间历史丢失（旺财不见）——SDK 对多条 user message 可能只重点保留首尾/最近，中间被稀释
   - **这才是"回归单条拼接"的真正理由**：不是 102_400 上限（那是 lead 的伪前提），而是**实测证明单条结构化文本上下文保留最完整**
3. **lead 的复盘**：① 该 spike 时跳过 spike（用静态分析骗自己），绕 3 轮 review；② 引入 `shouldQuery` 伪需求把简单问题搞复杂；③ 用未验证推断挡用户合理质疑。用户「传列表」直觉对，最佳实现是 1 条结构化文本。

## 对 plan 的最终影响（架构拍板）

- ✅ **单条 prompt 拼接架构正确**，但**理由更新**：从「102_400 上限被迫单条」改为「**实测证明单条结构化文本上下文保留最优**」。这是正向选择不是被迫妥协。
- ✅ **D3 拼接格式不变**（总结段 + 历史列表 + 当前消息，五等号块）——spike 方式A 的 `[历史回顾]\n用户:...\n助手:...` 正是此结构。
- ⚠️ **102_400 上限仍是真实约束**（单条 prompt 物理上限），但**绝大多数场景不触发**（前面 node 实算：30 条对话扣总结才占 ~95900 预算）。降级链（D6）/ 预算式拼接（不变量 6）是**边界兜底**不是主路径——保留但是 best-effort 心态，不是核心复杂度。
- ❌ **彻底放弃 shouldQuery / 多条独立 message** —— 实测都不如单条结构化文本。

## 残留风险 / 未验证

- 方式A 单条文本含超长 assistant 历史时仍可能撞 102_400 → 降级链（D6）兜底（预算式拼接列表逐条加到逼近预算停）。这是真实边界，但低频。
- codex SDK 对等行为未单独测（codex 注入走同款单条拼接，与 claude 对称，合理假设同行为；实施时 Step 8 实测验证）。

## 净结论（终于收敛）

**用户诉求「传历史列表」可行且简单 —— 实现就是把 DB 历史组织成 1 条结构化文本（`用户:.../助手:...`）拼进新会话首条 prompt。`shouldQuery` / 多条独立 message 是 lead 引入的伪复杂度，实测都更差。** 架构回归 deep-review R1-R3 打磨的单条拼接（总结 + 历史列表 + 当前消息），但心智从「被 102_400 逼的妥协」纠正为「实测最优选择」，cap/降级降级为低频边界兜底。3 个 spike（query 入口 / streamInput / 传历史）+ 7 变体实测背书。
